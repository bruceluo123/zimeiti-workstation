## 2. 数据契约（接入时按这个解析，别猜）

### 2.1 frontmatter
```yaml
---
tags: [标签1, 标签2]
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```
（旧素材可能没有 frontmatter，要容错：缺字段用文件 mtime / 文件名兜底。）

### 2.2 双链格式（图谱的边来自这里）
- `[[文件名]]` 或 `[[目录/文件名|显示名]]`
- 解析正则：`/\[\[([^\]|]+?)(?:\|([^\]]+))?\]\]/g`
  - 组1 = 链接目标（按 **basename 去 `.md`** 匹配，因为 Obsidian 按文件名解析，不认目录）
  - 组2 = 显示名（可空）
- **建图**：节点 = 每个 md 文件；边 = 文件内每条 `[[ ]]` 指向的目标。无向即可。
- **着色建议**：`maps/` = 枢纽（放大、亮色），`sources/` = 素材，`topics/entities/` 各一色。
- **中心节点**：`MOC_知识总图` 和 `骆奕州` 度数最高，做引力中心。

### 2.3 MOC 枢纽页内部结构（稳定，可解析出"板块卡片"）
每个 MOC 文件都含：一句话定位 → 分组的 `[[ ]]` 列表 → `## 相关枢纽` → `## 待合成（OUT 钩子）`（`- [ ]` 任务）。
你可以把"待合成"渲染成"该领域可产出的东西"待办区。

## 3. 接入方案（按推荐序）

### 方案 A — 最快上线（先嵌入，验证体验）
把 Quartz 产物当静态站，在工作站导航加「知识库」tab：
- 要么 `iframe` 指向独立部署的 Quartz；
- 要么把 Quartz `public/` 作为静态资源挂在工作站子路径。
- 成本：几乎为 0。缺点：风格和工作站不统一、导入闭环要另接。

### 方案 B — 原生融合（推荐终态）
新建路由 `src/app/knowledge/`（与现有 factory/flow/inspire/publish/studio 平级），并在 `src/components/layout` 的导航里加入口。
- **读数据**：构建时用 `gray-matter` 解析 frontmatter + 正文，`remark`/`react-markdown` 渲染。
- **三个视图**：
  1. **枢纽墙**：读 `maps/`，9 张卡片（用 §2.3 结构）。
  2. **关系图**：用 `react-force-graph-2d` 或 `d3-force`，数据按 §2.2 建图。
  3. **全文搜索**：`flexsearch` 或 `minisearch`，建在 build 期的 JSON 索引上。
- **建议新依赖**：`gray-matter`、`react-markdown` + `remark-gfm`、`react-force-graph-2d`、`minisearch`。
- **"在 Obsidian 中打开"按钮**（读网页、点回本地编辑）：
  `obsidian://open?vault=个人知识库&file=` + `encodeURIComponent("wiki/maps/MOC_知识总图")`

### 方案 C — 独立子域
Quartz 单独部署到子域名，工作站只放一个外链。适合不想耦合时。

> 我的建议：**先 A 验证观感，终态走 B。** B 才能和工作站的 Today/灵感/选题流打通（比如"复习卡片"推送、"素材→选题"一键带入 flow）。

## 4. 数据怎么从本地 markdown 流到部署环境

dev 时可直接读 `D:\wiki\个人知识库\wiki\`；但部署（Vercel）读不到本地盘。终态二选一：
1. **Git（推荐）**：`wiki/` 作为独立 repo / 或工作站的 git submodule；Obsidian 装 Git 插件自动提交；工作站构建时 `git pull` 拿最新 md。
2. **构建期拷贝**：CI 里把 `wiki/` 同步进工作站再 build（简单，但要自己管同步）。

```
贴 URL
```
